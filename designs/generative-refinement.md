# Generative Refinement of Distilled Feedback

## Problem

Distillation reads conversations between an owner and LLM, and produces a document that attempts
to capture the owner's judgment and mental models. That document can be used for subsequent prompts
to LLMs. In this project, we call that document a muse. The process has two operations (see
[grammar.md](../grammar.md)): `observe` extracts discrete observations from conversations, and
`update` folds those observations into the muse. Clustering, map-reduce, and incremental are
strategies for how `update` works; all share the same `observe` step and the same output type.

How do we know the muse is good? Currently, the owner reads it and decides. Quality is evaluated
by vibes — design documents, informal feedback about what the muse gets right or wrong, gut checks
on whether the text sounds like the person.

Prompt engineering is only measurable via vibes today. This makes the system of generating a muse
hard to reason about. A prompt change might improve one dimension and degrade another with no way
to tell without the owner re-reading the full output. The muse itself is a one-shot artifact —
whatever `update` produces, with no iterative refinement.

DSPy solves a similar problem for task-specific LLM programs: define a program, define a metric,
optimize the prompts to maximize the metric. The same structure applies here. The "program" is a synthetic conversation with context, where
we give the muse an opportunity to respond like the owner. The "metric" is an LLM judge scoring
alignment between those responses and what the owner actually said. The whole process is iterative
refinement: see what the muse says, measure the gap, take steps that move the prompt in a
measurably better direction.

## Core idea

Every conversation contains moments where the owner made a choice: pushed back on an approach,
corrected a design, expressed a preference, chose one option over another. If the muse captures
the owner, a model using it as system prompt should make similar choices at those moments.

```
conversation[0..n-1]  ->  muse + context  ->  predicted response
                                                    | judge
conversation[n]       ->  actual response  ->  similarity score
```

The gap between responses generated by the muse and responses the owner actually gave is a signal
for quality. This design proposes to use that signal to guide refinement.

## Solution

Generative refinement uses three operations: **curate**, **evaluate**, and **refine**.

**Curate** selects which moments in conversations are worth testing. Most user messages carry
little signal — we need the moments where the owner's response reveals how they think. Curate
takes a conversation and returns a set of replay examples, each pairing a conversation context
with the owner's actual response at a high-signal moment.

**Evaluate** measures how well a muse captures the owner. Given a muse and a set of replay
examples, it predicts what the owner would say at each moment, then judges how well those
predictions align with what the owner actually said. The output is a score.

**Refine** uses the evaluation signal to improve the muse. It analyzes where the muse scored
poorly, proposes edits to address those failures, and accepts or rejects the edits based on
whether the score improves.

### Curate

Most user messages reveal little. Saying "sounds good" to a good idea is not very informative.
We really only get information when the (muse, model, context) tuple produces something that
requires the owner to steer. Occurrences of steering are frequent today; we want to reduce them
over time, but until the muse is a perfect distillation of the owner, we'll need steering. The
curator scans conversations and identifies **high-signal moments** where the owner's response
would differ from a generic competent person's.

Signal markers:
- **Course corrections**: "No, do it this way instead" — the owner rejected the model's approach
- **Architectural decisions**: choosing between options with real tradeoffs
- **Explicit preferences**: "I prefer X over Y" with reasoning
- **Opinionated feedback**: "This is too complex" or "I don't love that"
- **Reasoning under uncertainty**: hedging, distinguishing knowledge from guesses

The curator is a Sonnet-tier call: mechanically compressed conversation in (code blocks stripped,
tool output collapsed — same compression as `observe`), message indices worth evaluating out, with
a rationale per selection. The rationale is for debugging, and eventually observability and user
intervention.

Output per conversation: **replay examples**.

```go
type ReplayExample struct {
    ConversationID string
    Source         string
    Context        []Message  // conversation up to the prediction point
    Target         Message    // what the owner actually said
    Signal         string     // why this moment was selected (debugging)
}
```

Curation is cached per conversation because we believe a low-temperature call to Sonnet with a
given conversation will yield consistent results. This is worth testing.

**Training and testing data.** Examples are split by conversation, not by message — all examples
from a given conversation go into the same split. This prevents information leakage from earlier
messages in the same conversation. Default: 80/20 training/testing by conversation count,
deterministic by conversation ID hash.

### Evaluate

Evaluation runs the muse against each replay example and scores the result.

Per example:
1. Construct a prompt: the muse system prompt + mechanically compressed conversation context
2. Call the model (Sonnet-tier) to predict the owner's response
3. Call a judge model (Sonnet-tier) to score the prediction against the actual response

The judge scores on dimensions that matter for a muse, not for general text similarity:

- **Priorities**: Does the prediction focus on the same things the owner focused on?
- **Reasoning**: Does it arrive at similar conclusions through similar logic?
- **Disposition**: Does it push back where the owner pushed back, accept where they accepted?
- **Taste**: Does it reflect the same aesthetic and stylistic preferences?

Each dimension is judged as **good**, **bad**, or **unsure**. Not a 1-5 scale — the same problem
Netflix found with star ratings applies here. A 3 vs. 4 distinction is hard for a judge to make
consistently, so the coarser signal is more reliable in aggregate. The metric is the percentage of
"good" judgments across dimensions and examples.

```go
type Judgment string  // "good", "bad", "unsure"

type EvalScore struct {
    ExampleID   string
    Priorities  Judgment
    Reasoning   Judgment
    Disposition Judgment
    Taste       Judgment
}

type EvalResult struct {
    MuseFingerprint    string
    Scores             []EvalScore
    GoodRate           float64  // fraction of "good" across all judgments
    TrainingGoodRate   float64  // fraction on training data only
    TestingGoodRate    float64  // fraction on testing data only
}
```

Evaluation is the expensive step. Two Sonnet calls per example (predict + judge), scaling linearly.
~50 conversations yield ~100-200 curated examples: 200-400 Sonnet calls per round, multiple rounds
per refinement run. For comparison, clustered distillation on 50 conversations uses ~50 Sonnet
calls for observation and ~50 for labeling, plus one Opus call for composition. A single evaluation
round costs 2-4x what a full distillation costs, and refinement needs multiple rounds.

**Cost control.** Training data for refinement, testing data at the end. Within rounds, we
subsample — random subset per round, full training set periodically.

### Refine

The refinement loop takes an initial muse and iteratively improves it against the evaluation
metric. The muse.md is the prompt. The loop edits it to improve the metric.

```
muse(0)    = output of distillation pipeline
muse(k+1)  = refine(muse(k), failures(k))
```

Each round:
1. **Evaluate** the current muse on a sample of training examples
2. **Analyze failures**: select the N lowest-scoring examples. For each, surface the predicted
   response, the actual response, and the judge's reasoning for the low score.
3. **Propose one edit**: an Opus call takes the current muse and the failure analysis, and
   produces a single coherent diff. The diff may touch multiple sections but is one atomic
   change. The prompt constrains edits to be conservative — add or refine, don't wholesale
   rewrite — and requires justification with reference to specific failures.
4. **Apply and re-evaluate**: apply the diff, evaluate the updated muse on the same sample plus
   a fresh sample (to detect overfitting to the subsample).
5. **Accept or reject**: if the new muse scores higher on the combined sample, accept. Otherwise,
   reject and move on to the next round.

Convergence: improvement drops below threshold, or fixed budget (default 5 rounds).

```
distill -> muse(0) -> evaluate -> analyze failures -> propose edits -> muse(1) -> ...
                          ^                                                |
                          +------------------------------------------------+
```

After refinement completes, the evaluation results are a natural feedback surface. The owner can
review failure examples — where the muse diverged most from what they actually said — and decide
whether the muse or the metric is wrong. This is an opportunity for human-in-the-loop correction
before the next refinement run.

### CLI

```bash
muse refine                            # run refinement loop on current muse
muse refine --rounds=5                 # set refinement budget
muse refine --eval-only                # evaluate without refining
muse refine --curate                   # re-curate replay examples only
```

The refine command depends on a muse existing — either from a prior `muse distill` or from a
previous refinement run. The pipeline is: `distill` produces muse(0), `refine` improves it.

### Storage

```
~/.muse/
├── distill/
│   └── ...                                          # existing pipeline artifacts
├── refine/
│   ├── examples/{source}/{conversation_id}.json     # curated replay examples
│   ├── evals/{timestamp}.json                       # evaluation results per round
│   └── proposals/{timestamp}.json                   # proposed edits per round
├── muse/versions/{timestamp}/muse.md                # output (same as distill)
```

Refinement artifacts live under `refine/` parallel to `distill/`. The muse output goes to the
same versioned location — a muse produced by distillation and one improved by refinement are
indistinguishable from the consumption side.

## Decisions

### Why LLM-as-judge over text similarity metrics?

BLEU, ROUGE, and embedding cosine distance measure surface-level textual overlap. A good muse
might express the owner's reasoning in different words. Matching specific words is less important
than capturing the correct conclusions for the correct reasons. LLM-as-judge evaluates semantic
alignment on the dimensions that matter (priorities, reasoning, disposition, taste).

The judge is noisy. Aggregating across examples and dimensions averages out per-example variance.
Human feedback would be more reliable, but human feedback is expensive.

### Why strategic message selection over all messages?

Most user messages are low-information: confirmations, short instructions, routine approvals.
Evaluating on all of them dilutes the signal (most predictions trivially match because the
baseline is "sounds good") and wastes budget. Strategic selection focuses on the moments that
differentiate this person from a generic user.

### Why split by conversation, not by message?

Messages within a conversation are not independent. If the refinement loop sees the owner's
correction at turn 10, it leaks information about what the owner would say at turn 15. Splitting
by conversation ensures the testing data contains genuinely unseen reasoning patterns.

### Why conservative edits over wholesale rewrite?

The muse from distillation encodes information from every observed conversation. Wholesale
rewriting risks losing patterns absent from the evaluation sample. Conservative edits (add,
refine, adjust emphasis) preserve what's there and only modify what the evaluation identifies as
wrong or missing. Same "slow learning rate" principle as incremental distillation.

### Why Opus for edit proposals, Sonnet for everything else?

Prediction and judging are focused tasks Sonnet handles well. Proposing edits requires reasoning
about a complex document, weighing changes against preservation — editorial judgment that benefits
from Opus. Matches the existing tier split: Sonnet for bulk work, Opus for editorial judgment.

### Why one atomic edit per round?

An edit may touch multiple sections of the muse, but it is proposed and evaluated as a single
diff. Three alternatives were considered:

**Structured batch with greedy application.** Opus proposes N discrete edits, each tied to a
specific failure. Apply them one at a time in failure-severity order, evaluate after each, keep the
ones that help. Finer signal, but N times the evaluation cost per round, and order-dependent —
edit A might only help when edit B is already applied, but if A is evaluated first and rejected,
we never discover the combination.

**Batch then prune.** Apply all N edits, evaluate the batch. If the batch helps, remove each edit
one at a time to find which ones improved the score. Principled, but N+1 evaluations per round.

**Propose whole-muse alternatives.** Generate 2-3 candidate muse variants, each taking a different
angle on the failures. Evaluate all, keep the best. Closer to how DSPy's MIPRO works, but loses
the incremental property — each variant is a near-complete rewrite, which risks dropping patterns
not in the evaluation sample.

One atomic edit per round is the simplest approach that gives a clean accept/reject signal. If
rounds oscillate (one part of the diff helps, another hurts, net zero), that's the signal to move
to greedy or pruning.

### Why 5 rounds as a default?

Diminishing returns. The first round captures obvious gaps — missed patterns, wrong emphasis.
Subsequent rounds pick up subtler issues. By round 5, edits are marginal. More rounds available
via `--rounds`.

### Why not refine from scratch (skip distillation)?

Distillation processes every conversation and extracts patterns systematically. Starting from empty
would require the refinement loop to both discover patterns and refine their expression — a harder
search problem. Starting from distillation output lets the loop focus on refinement: adjusting
emphasis, adding nuance, removing false patterns.

### Why not fine-tune instead of prompt refinement?

Fine-tuning would train a model to behave like the owner directly, bypassing the muse document
entirely. This has two problems. First, the muse document is a readable artifact that the owner
can inspect, edit, and share — fine-tuned weights are opaque. Second, fine-tuning requires
infrastructure (training runs, model hosting) that prompt refinement doesn't. The muse is
portable across any model that accepts a system prompt.

## Deferred

### Few-shot exemplar selection

The muse captures patterns abstractly. Few-shot examples ground them in concrete behavior: real
conversation snippets that demonstrate the owner's patterns. These can be augmented with human
feedback — the owner selects or edits exemplars that best represent how they think.

For each candidate exemplar (drawn from curated high-signal moments), test whether including it
alongside the muse improves scores on other examples. Exemplars that help across multiple cases
are good; ones that help on only one are overfit.

Exemplars are appended to the muse as a `## Examples` section. Budget: ~5-10 exemplars, enough to
ground the abstract patterns without bloating the system prompt. Selection is greedy: start with
the best single exemplar, add the one that most improves the metric given the current set, repeat
until marginal improvement drops below threshold or the budget is hit.

### Pipeline stage prompt refinement

Refine the prompts used in `observe` and `update` so the pipeline produces a better muse as
measured by the replay metric. Changing a stage prompt changes the muse, which changes the score.

Expensive — each iteration requires a full or partial pipeline run. Deferred until direct muse
refinement demonstrates the replay metric is useful.

### Dimension weighting

The four judge dimensions (priorities, reasoning, disposition, taste) are equally weighted. Some
owners may care more about reasoning fidelity than stylistic taste, or vice versa. Custom
weighting is straightforward to add. **Revisit when:** users report that the refinement loop
improves on dimensions they care less about at the expense of ones they care more about.

### Multi-model evaluation

The evaluation currently uses one model for prediction and one for judging. Different models might
expose different gaps in the muse — a muse that works well with Sonnet might fail with a different
model. **Revisit when:** the muse is used across multiple model families and cross-model
consistency matters.

### Active learning for curation

The curator selects examples statically. An active learning variant would select examples the
current muse gets most wrong, focusing budget where it's most informative. **Revisit when:**
evaluation cost becomes the bottleneck.

## Deferred until agreement on this design

### Grammar updates

This design introduces three operations not in [grammar.md](../grammar.md): `curate`, `evaluate`,
and `refine`. The grammar should be updated to include them, along with a composite `refine`
operation. Blocked on agreement that this design is worth building.
